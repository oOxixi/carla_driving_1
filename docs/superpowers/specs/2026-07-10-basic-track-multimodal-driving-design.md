# 基础赛道多模态语音驾驶系统设计

日期：2026-07-10
状态：用户已批准，进入实施计划阶段
适用范围：CARLA 0.9.16 基础赛道，单人开发，两周周期，单机离线运行，RTX 5070 Laptop GPU 8GB

## 1. 背景与目标

系统用于验证自然语言语音指令在 CARLA 智能驾驶典型场景中的可用性。语音组负责把音频转换为文本；车辆控制组负责文本之后的全部链路，包括场景构建、传感器数据采集、感知、多模态语义对齐、行为决策、车辆控制、安全兜底和自动评测。

基础赛道首期交付目标是一个可以离线、稳定、重复运行的最小闭环，覆盖基础操控、复杂避障和应急响应三类场景。系统优先保证安全性、任务完成率和可测量性，不追求通用自动驾驶能力。

内部验收目标如下：

- 场景任务完成率不低于 95%，高于赛题要求的 90%。
- 多模态语义目标对齐准确率不低于 99%，高于晋级参考线的 98%。
- 常见指令的车辆组解析耗时不高于 40ms。
- 从收到 ASR 结果到输出可执行高层决策的车辆组链路 P95 延迟不高于 120ms，为全流程 150ms 留出接口与调度余量。
- 所有标准场景均支持至少连续运行 20 次并自动统计结果。
- 安全规则具有最高优先级；模型超时、非法输出和低置信度输入不会导致车辆继续危险行驶。

语音识别准确率由语音组负责。车辆组负责保留 ASR 原始文本、置信度和时间戳，以便联合计算端到端指标。

## 2. 范围与非目标

### 2.1 本期范围

- CARLA 同步仿真、传感器和交通参与者管理。
- RGB 摄像头、LiDAR、车辆状态和天气数据的同步采集。
- 基于预训练轻量模型的图像目标检测。
- 基于几何方法的 LiDAR 感兴趣区域过滤、地面过滤和障碍物聚类。
- 图像目标与点云距离的轻量关联。
- 语音文本、视觉目标、点云目标、车辆状态和天气状态的统一场景表示。
- 常见指令快速解析与复杂/模糊指令的本地小型多模态模型处理。
- 受限动作空间、高层决策、安全仲裁、路径规划和 PID 控制。
- 六个标准场景、统一数据记录和自动化评测。
- NuScenes、Waymo 代表性子集与 CARLA 数据的统一记录格式适配。
- 引入与 CARLA 0.9.16 精确匹配的官方 ScenarioRunner，复用场景编排、触发条件和评测标准。
- 定向研究 LMDrive 的语言驾驶架构、指令分类和多模态数据格式，并转换为本项目的公共 schema。
- 定向研究 DriveMLM 的“多模态行为决策 + 传统运动规划”边界，并将其行为规划状态映射到本项目动作空间。

### 2.2 本期非目标

- 从零训练视觉基础模型或端到端自动驾驶大模型。
- 覆盖任意地图、任意自然语言和开放世界驾驶。
- 逐帧调用大模型并由大模型直接输出油门、刹车或转向值。
- 全量训练 NuScenes 或 Waymo 数据集。
- 自研剪枝、量化、知识蒸馏和车规芯片算子适配。
- Thor-U、地平线 J6、CPU/GPU/NPU/DSP 异构调度和 HIL 测试。
- 使用 CARLA 目标真值替代正式感知输入。

可以使用现成的低比特推理权重以适应 8GB 显存，但本期不把量化方法本身作为研究交付。

## 3. 责任边界

### 3.1 语音组

语音组输入音频，输出最终 ASR 文本、识别置信度和统一时间戳。语音组负责普通话、方言和轻微噪声下的识别准确率。

语音组不直接输出油门、刹车、转向或变道轨迹，也不负责将“前面的行人”等语言指代关联到仿真目标。

### 3.2 车辆控制组

车辆控制组接收 ASR 结果，完成：

1. CARLA 场景、天气、车辆、行人和障碍物生成。
2. 摄像头、LiDAR、车辆状态和环境状态采集。
3. 目标检测、点云处理和轻量传感器融合。
4. 自然语言指令解析与场景目标关联。
5. 行为决策、安全校验、路径规划和车辆控制。
6. 数据集记录、延迟埋点、任务判定和评测报告。

## 4. 总体架构

系统采用分层混合架构：轻量感知高频运行，多模态模型按指令或关键事件触发，确定性安全模块和控制器持续闭环。

```text
音频
  │
  ▼
语音组 ASR ── ASRCommand ───────────────────────────┐
                                                    │
RGB 摄像头 ── 图像检测 ───────┐                     │
                               ├─ SceneState ────────┼─ 语义对齐/融合决策
LiDAR ─────── 点云聚类/测距 ──┤                     │
                               │                     │
车辆状态/天气/地图 ───────────┘                     │
                                                    ▼
                                             DrivingIntent
                                                    │
                                                    ▼
                                               安全仲裁器
                                                    │
                                                    ▼
                                           路径规划 + PID 控制
                                                    │
                                                    ▼
                                                CARLA 车辆
                                                    │
                                                    ▼
                                               指标与日志
```

大模型只生成高层、受约束的驾驶意图，不直接控制车辆执行器。紧急制动、碰撞风险判断、超时处理和最低风险停车由确定性安全模块负责。

## 5. 组件设计

### 5.1 仿真与传感器管理

CARLA 使用同步模式，基础配置为：

- 固定仿真步长 0.05 秒，对应 20Hz。
- RGB 摄像头默认 640×384、10Hz。
- LiDAR 默认 10Hz，并限制有效距离和点数以控制负载。
- 车辆状态以 20Hz 采集。
- 采用低画质或等效低负载配置，为推理保留显存。

传感器数据按 CARLA frame ID 和单调时钟对齐。融合模块只消费时间差在允许窗口内的数据；过期帧不得参与决策。

场景编排优先采用官方 [CARLA ScenarioRunner](https://github.com/carla-simulator/scenario_runner) `v0.9.16`，确保与当前 CARLA 版本一致。ScenarioRunner 负责参与者生成、行为触发、天气、场景超时和成功/失败条件；本项目负责 ego agent、传感器、语义对齐、决策与控制。第一阶段重点验证并复用行人横穿、施工障碍、静态/高速加塞、天气变化、碰撞判定和路线完成条件，不重复实现已有的场景基础设施。

### 5.2 感知

摄像头使用轻量预训练检测器，首期支持车辆、行人、交通灯及道路常见障碍物。LiDAR 使用几何处理获得障碍物点簇、方位和距离。锥桶等视觉模型不稳定的目标可由点云“未知障碍物”补充，但正式决策不能读取 CARLA actor 真值。

融合采用可解释的轻量关联：根据相机标定把点云投影到图像平面，使用检测框内的稳健距离统计给视觉目标补充距离；没有视觉类别的近距离点簇保留为 unknown_obstacle。

感知输出包含稳定目标 ID、类别、方位、距离、相对速度、置信度和风险等级。首期不要求复杂的全场景多目标跟踪，只要求在一次指令执行窗口内维持稳定关联。

### 5.3 统一场景表示

`SceneState` 是感知、语义对齐和决策之间的唯一共享输入。它包含：

- 当前帧号和时间戳。
- 自车位置、速度、航向、转向、当前车道和路线状态。
- 天气、昼夜和道路条件。
- 可见目标列表及目标 ID。
- 可变道方向、前向碰撞时间和当前风险等级。
- 最近一个可供多模态模型查看的关键 RGB 帧引用。

模块通过数据结构通信，避免直接依赖其他模块内部实现。

### 5.4 指令解析与多模态语义对齐

系统采用两级处理：

1. 快速路径处理启动、停车、加减速、转向、明确方向变道等高频命令。它将文本转换成受限动作和参数，目标是稳定满足解析延迟要求。
2. 复杂路径处理组合动作、模糊应急指令和场景指代。候选实现为可在 8GB 显存内运行的 3B 级现成 4-bit 多模态模型。输入为 ASR 文本、关键 RGB 帧和序列化 `SceneState`；输出必须符合固定 JSON schema。

复杂指令先提取动作、目标描述、空间关系、顺序关系和目标速度，再从 `SceneState.objects` 选择候选目标。比如“前面的行人”依次按类别、相对方位、可见性、距离和风险排序，最终绑定到稳定目标 ID。

多模态模型是复杂避障和应急响应场景的正式决策组成部分，不是仅用于展示的旁路。快速路径只承担明确基础命令和模型失效时的安全降级，不能用固定脚本替代复杂场景中的多模态对齐与决策。

如果没有唯一候选目标、模型置信度不足或场景信息过期，系统不得猜测执行，而应进入安全减速或请求确认状态。

语言驾驶的数据组织重点参考 [LMDrive](https://github.com/opendilab/LMDrive)，但不安装其 CARLA 0.9.10.1、LAVIS、7B 模型和旧 Python 运行栈。需要研究并转化的内容限于：导航指令与驾驶提醒的分类、语言指令和传感器片段的时间绑定、多视角图像/LiDAR/车辆状态记录格式，以及语言指令到 agent controller 的接口。转化结果必须落到本项目的 `ASRCommand`、`SceneState` 和 `DrivingIntent`，不能引入第二套并行接口。

### 5.5 行为决策

模型输出限制在以下动作集合：

- `START`
- `STOP`
- `SET_SPEED`
- `TURN_LEFT`
- `TURN_RIGHT`
- `CHANGE_LANE_LEFT`
- `CHANGE_LANE_RIGHT`
- `AVOID_OBJECT`
- `EMERGENCY_BRAKE`
- `RETURN_TO_LANE`

组合指令表示为有顺序和完成条件的动作列表。例如“绕开前方行人后提速到 20km/h”会生成 `AVOID_OBJECT` 和后继 `SET_SPEED`，只有避障完成后才允许进入后继动作。

行为决策边界重点参考 [DriveMLM](https://github.com/OpenGVLab/DriveMLM)：多模态模型负责把用户指令、传感器语义和驾驶规则对齐到有限的行为规划状态，传统规划器负责将行为状态转换为轨迹和控制量。DriveMLM 当前不作为代码或模型依赖；其研究产出是行为状态映射、模型输入输出约束和解释字段设计。若仓库缺少可运行实现，不得为复现论文消耗基础赛道工期。

### 5.6 安全仲裁

安全仲裁器优先级高于语音指令和模型输出。它负责：

- 根据前向距离、相对速度和碰撞时间触发减速或紧急制动。
- 在目标车道被占用时阻止变道。
- 在模型输出非法、超时或低置信度时触发最低风险停车。
- 在语音指令与当前道路风险冲突时覆盖指令。
- 防止动作长时间不结束，并将超时标记为任务失败。

### 5.7 规划与控制

全局导航复用 CARLA 地图和 `GlobalRoutePlanner`。局部轨迹通过道路 waypoint、目标车道和避障状态生成。纵向与横向执行复用或封装 CARLA PID 控制能力。

变道、绕障和回归车道使用有限状态机，不依赖大模型逐帧重规划：

```text
检查安全间隙 → 减速 → 离开原车道 → 通过障碍 → 回归车道 → 恢复目标速度
```

## 6. 核心接口

### 6.1 ASR 输入

```json
{
  "command_id": "cmd_001",
  "text": "绕开前面的行人后加速到20公里每小时",
  "confidence": 0.97,
  "timestamp_ms": 123456789
}
```

缺少文本、置信度或时间戳的消息视为无效输入。重复 `command_id` 不得重复执行。

### 6.2 场景状态

```json
{
  "frame_id": 1024,
  "timestamp_ms": 123456800,
  "ego": {
    "speed_kmh": 12.4,
    "lane_id": 1
  },
  "weather": {
    "rain": 0.0,
    "night": false
  },
  "objects": [
    {
      "id": "pedestrian_12",
      "type": "pedestrian",
      "direction": "front",
      "distance_m": 13.2,
      "relative_speed_mps": -1.1,
      "confidence": 0.93,
      "risk": "high"
    }
  ]
}
```

### 6.3 驾驶意图

```json
{
  "command_id": "cmd_001",
  "actions": [
    {
      "action": "AVOID_OBJECT",
      "target_id": "pedestrian_12"
    },
    {
      "action": "SET_SPEED",
      "target_speed_kmh": 20
    }
  ],
  "confidence": 0.94,
  "reason": "front pedestrian matched by class, direction and distance"
}
```

生产控制逻辑只读取 schema 中的受限字段，不执行模型自由文本。

## 7. 标准场景

系统首期交付六个可重复场景：

1. **基础启停**：车辆启动、加速到指定速度、减速并停车。
2. **转向与变道**：执行左/右转以及明确方向的安全变道。
3. **行人避障**：识别并绕开前方行人，完成后提升到指定速度。
4. **锥桶避障**：识别施工锥桶或未知近距离障碍，避让后回归原车道。
5. **恶劣环境停车**：雨天或夜间收到模糊危险指令后减速停车。
6. **突发加塞**：前车突然切入时优先紧急制动；只有存在明确安全空间时才允许避让。

每个场景包含固定随机种子、初始位置、天气、交通参与者、语音文本集合、成功条件、失败条件和最大执行时间。

## 8. 数据设计

CARLA、NuScenes、Waymo 和语音数据统一到同一条记录结构：

```text
sample_id
scene_id
timestamp/frame_id
ASR 文本、置信度和音频引用
RGB 图像引用
LiDAR 点云引用
车辆状态
天气状态
感知真值与目标关联真值
期望高层动作
实际高层动作
任务结果
各阶段延迟
```

CARLA 是首期主要训练与测试数据来源。NuScenes 和 Waymo 只选取与车辆、行人、锥桶/道路障碍、雨夜和切入场景相关的代表性子集，完成统一格式适配和基础验证，不进行全量重训练。语音组提供普通话、方言和轻微噪声版本；车辆组标注文本对应的场景目标和期望动作。

训练集与测试集按场景实例和随机种子隔离，避免同一仿真轨迹同时出现在训练和测试中。

## 9. 性能与资源预算

单机 8GB 显存按以下原则分配：

- CARLA 使用低画质和低分辨率传感器，目标显存占用约 3–4GB。
- 轻量感知目标显存占用约 1–2GB。
- 多模态模型使用现成低比特权重并按事件触发；必须通过实测决定是否与感知同时驻留。
- 若同时驻留超出预算，优先采用模型串行调度、降低输入分辨率或将非关键预处理放到 CPU；不得通过关闭安全感知换取显存。

所有阶段记录单调时钟：ASR 接收、指令解析开始/结束、感知帧时间、融合结束、决策输出、安全仲裁完成和控制生效。评测同时报告平均值、P95 和最大值，不能只报告最佳样本。

## 10. 失败处理

- ASR 置信度不足：不执行危险动作，进入安全减速或请求确认。
- 指令无法解析：返回明确错误并保持或进入安全状态。
- 指代目标不存在或不唯一：不猜测目标，减速并等待确认。
- 感知帧过期：拒绝基于旧场景执行变道或绕障。
- 多模态模型超时：取消该结果并触发最低风险停车。
- 模型 JSON 不符合 schema：视为非法输出，不进入控制模块。
- CARLA 连接断开或传感器丢帧：停止任务计时，安全结束本轮并记录基础设施故障。
- 规划无解：停车并记录失败原因，不持续重复危险规划。

## 11. 测试与验收

### 11.1 单元测试

- ASR 消息校验、去重和超时。
- 中文速度、方向、顺序关系和否定表达解析。
- 空间指代目标排序与唯一性判断。
- 决策 JSON schema 校验。
- 安全规则优先级和模型非法输出拦截。
- 场景完成条件和延迟计算。

### 11.2 集成测试

- 传感器帧同步与过期数据拒绝。
- 相机检测与 LiDAR 距离关联。
- ASRCommand 到 DrivingIntent 的完整链路。
- DrivingIntent 经安全仲裁、规划和 PID 后驱动车辆。
- 多模态模型超时、崩溃和低置信度降级。

### 11.3 场景回归

六个标准场景分别使用固定随机种子连续运行至少 20 次。记录任务成功、碰撞、越线、超时、错误目标关联、最低距离以及各阶段延迟。演示版本只有在没有阻断级安全问题且内部指标达标后才能冻结。

## 12. 两周交付顺序

1. 第 1 天：引入并固定 ScenarioRunner `v0.9.16`，完成 CARLA 连接、同步模式和一个官方场景冒烟测试。
2. 第 2 天：复用 ScenarioRunner 运行行人/障碍或加塞场景；完成 RGB、LiDAR、车辆状态同步与日志记录。
3. 第 3 天：路线规划、PID 控制和紧急停车基线。
4. 第 4–5 天：图像检测、点云处理和轻量融合。
5. 第 6 天：统一 `SceneState` 和稳定目标 ID。
6. 第 7 天：基于 LMDrive 的指令分类和片段格式完成 ASR JSON 接口、数据转换与快速指令解析。
7. 第 8 天：按 DriveMLM 行为规划边界接入本地多模态模型，完成受限 JSON 输出和实测资源评估。
8. 第 9 天：语义目标对齐、安全仲裁和降级路径。
9. 第 10 天：基础启停、转向和变道场景。
10. 第 11 天：行人和锥桶避障场景。
11. 第 12 天：雨夜危险和突发加塞场景。
12. 第 13 天：数据适配、自动评测和延迟优化。
13. 第 14 天：回归测试、代码冻结、录屏和材料整理。

如果进度落后，保留顺序为：安全闭环、六个场景、自动评测、多模态模型、外部数据适配。不得牺牲安全闭环换取额外模型或数据集展示。

## 13. 目录与模块边界建议

```text
src/
  interfaces/     # ASRCommand、SceneState、DrivingIntent schema
  simulator/      # CARLA 连接、场景、传感器和同步
  perception/     # 图像检测、点云处理和融合
  alignment/      # 指令解析、目标指代和多模态模型适配
  decision/       # 动作编排和行为状态机
  safety/         # 风险判断、仲裁和最低风险停车
  planning/       # 路线和局部轨迹
  control/        # PID 与 CARLA VehicleControl
  evaluation/     # 指标、日志和场景判定
scenarios/        # 六个标准场景配置
tests/            # 单元、集成和场景回归测试
scripts/          # 启动、数据采集、评测和演示入口
configs/          # 仿真、传感器、模型和阈值配置
external/         # 固定版本的外部运行依赖；首期仅允许 ScenarioRunner
docs/references/  # LMDrive、DriveMLM 等只读研究记录和映射结论
```

每个模块只依赖公共接口，不跨模块读取内部状态。模型、感知器和控制器均通过适配器封装，以便挑战赛道替换为轻量化实现。

## 14. 主要风险与缓解措施

- **显存不足**：前两天实测 CARLA 基线占用；降低分辨率并采用事件触发模型，不让大模型逐帧运行。
- **多模态模型延迟超标**：常见指令走快速路径；模型只处理复杂语义，并限制输入长度、图像分辨率和输出 token。
- **锥桶识别不稳定**：视觉类别检测与 LiDAR unknown obstacle 双通道兜底。
- **端到端指标定义不清**：与语音组统一时间戳，分别报告 ASR、车辆组决策和完整链路延迟。
- **外部数据适配耗时**：只处理代表性子集并统一格式，不做全量训练。
- **单人开发范围失控**：以六个标准场景和自动回归为冻结边界，其他能力延后。
- **旧项目环境污染**：不安装 LMDrive 的旧版 CARLA、Python、LAVIS 或 7B 依赖；只做源码阅读和格式映射。
- **DriveMLM 可运行资产不足**：只采用其模块边界和行为状态思想，不等待未发布或不可用的代码与权重。
- **第三方代码许可证和归属不清**：固定依赖版本，保留 LICENSE、来源链接和修改记录；ScenarioRunner、LMDrive、DriveMLM 分别按其上游许可证要求处理。

## 15. 后续挑战赛道兼容性

本期不实现车规部署，但保留以下兼容点：

- 模型通过统一推理适配器调用。
- 公共输入输出使用稳定 schema。
- 每个阶段有独立延迟和资源埋点。
- 感知、语义模型和决策模型可以分别替换。
- 数据记录能够支持后续蒸馏、剪枝和量化校准。

这些边界允许后续在不重写仿真、场景、安全和控制模块的情况下替换为车规级轻量模型。

## 16. 重点开源方案考察与采用决策

本期只把一个外部项目作为运行依赖，把两个项目作为架构与数据研究对象，避免三套旧环境同时进入主工程。

| 项目 | 定位 | 重点考察 | 预期产出 | 采用门槛 | 停止条件 |
|---|---|---|---|---|---|
| [CARLA ScenarioRunner](https://github.com/carla-simulator/scenario_runner) | 直接运行依赖 | 行人、施工障碍、加塞、天气、碰撞、路线完成和超时评测 | 固定 `v0.9.16` 的依赖接入；至少三个官方场景冒烟测试；六个比赛场景的复用映射表 | Windows 下可连接现有 CARLA 0.9.16；同步模式稳定；ego agent 可由本项目控制 | 单个最小场景经过半天排查仍无法运行时，保留其场景定义和评测逻辑，改为最小兼容封装 |
| [LMDrive](https://github.com/opendilab/LMDrive) | 架构和数据格式参考 | 导航指令/驾驶提醒分类、多视角图像与 LiDAR 记录、语言和驾驶片段绑定、agent controller 接口 | `LMDrive → 本项目 schema` 映射文档；指令分类表；可导入的小型示例数据 | 不安装旧运行栈；映射可由当前 Python 环境单独测试；不新增重复数据结构 | 研究超过半天仍不能产生可验证映射时，只保留论文级架构引用 |
| [DriveMLM](https://github.com/OpenGVLab/DriveMLM) | 行为决策边界参考 | MLLM 输入、行为规划状态、解释字段、与传统运动规划器的分工 | `DriveMLM behavior state → DrivingIntent` 映射；受限 JSON schema；模型和规划器职责清单 | 产出能够直接约束本地多模态模型输出；不依赖未发布资产 | 缺少代码或权重不视为阻塞；完成架构映射后立即停止复现工作 |

### 16.1 ScenarioRunner 验收证据

ScenarioRunner 的接入必须留下以下可重复证据：

- 上游地址、`v0.9.16` 标签和实际固定的提交哈希。
- Windows 启动命令、CARLA 端口、Python 版本和环境变量记录。
- 行人、障碍/施工、加塞三类场景至少各运行一次的结果文件。
- 天气切换、碰撞判定、路线完成和超时判定的自动化输出。
- 本项目 ego agent 接管控制后仍能复用场景触发与判定。

### 16.2 LMDrive 研究问题

LMDrive 研究只回答以下问题：

1. 如何区分导航指令、驾驶提醒和组合指令？
2. 一条语言指令如何与 2–20 秒的多模态驾驶片段绑定？
3. 多视角图像、LiDAR、车辆状态、路线和控制信号如何组织在同一数据样本中？
4. 哪些字段可以直接映射到 `ASRCommand`、`SceneState` 和 `DrivingIntent`？
5. 哪些旧实现依赖 CARLA 0.9.10.1 或 7B 模型，必须明确排除？

研究结束标志是形成映射文档和一个无需 LMDrive 环境即可读取的最小转换样例，而不是成功运行 LMDrive 全模型。

### 16.3 DriveMLM 研究问题

DriveMLM 研究只回答以下问题：

1. 多模态模型应该输出哪些行为规划状态，而不是连续控制量？
2. 用户指令、驾驶规则、视觉/LiDAR语义和车辆状态如何进入同一决策请求？
3. 行为规划状态怎样映射到本项目有限状态机、局部规划和 PID？
4. 决策解释字段怎样用于调试和评测，而不被控制器当作可执行文本？
5. 安全仲裁器在哪些条件下覆盖模型决策？

研究结束标志是形成受限行为 schema、状态映射和一组契约测试。DriveMLM 仓库是否提供完整可运行模型不影响本项目进度。

### 16.4 时间预算

三项考察的总预算不超过两天，并与系统基线并行落地：

- ScenarioRunner：最多 1 天，目标是形成真实运行依赖。
- LMDrive：最多半天，目标是形成数据和指令映射。
- DriveMLM：最多半天，目标是形成行为决策边界和 schema。

任何考察超过预算且没有产生可执行接口、测试或映射文档时，立即停止继续复现，把时间返还给六个标准场景和安全闭环。
